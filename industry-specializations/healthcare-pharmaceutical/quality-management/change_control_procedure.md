## 1. Introduction

### 1.1 Purpose

This procedure defines the processes and responsibilities for the systematic evaluation, approval, implementation, and documentation of changes that may affect quality, safety, efficacy, regulatory compliance, or GxP operations. An effective change control system ensures that:

- Changes are properly evaluated for impact before implementation
- Changes are properly documented, reviewed, and approved
- The potential risks associated with changes are identified and mitigated
- Appropriate validation and verification activities are performed
- Regulatory compliance is maintained throughout the change process
- Affected stakeholders are informed of changes and receive appropriate training
- Changes are implemented in a controlled manner to minimize disruption

### 1.2 Scope

This procedure applies to all changes that may affect:

- Product quality, safety, and efficacy
- Validated systems, facilities, equipment, and processes
- Raw materials, components, and packaging materials
- Manufacturing and testing methods
- Product specifications and analytical methods
- Documentation including procedures, records, and batch documentation
- Computerized systems and data handling processes
- Organizational structure and responsibilities

This procedure applies to all facilities, departments, and functions involved in GxP activities, including but not limited to:

- Manufacturing
- Quality Control
- Quality Assurance
- Engineering and Maintenance
- Regulatory Affairs
- Research and Development
- Clinical Operations
- Supply Chain
- Information Technology

### 1.4 Definitions

**Change**: Any modification to facilities, systems, equipment, processes, materials, specifications, documentation, software, or organizational structures that may affect product quality, safety, efficacy, or regulatory compliance.

**Change Control**: A formal process used to ensure that changes to a product, system, or process are introduced in a controlled and coordinated manner to minimize the risk of unintended consequences.

**Change Request**: A formal document initiating the change control process, detailing the proposed change, its justification, and preliminary impact assessment.

**Impact Assessment**: A systematic evaluation of the potential effects of a proposed change on product quality, safety, efficacy, regulatory compliance, and validated systems.

**Risk Assessment**: A systematic process to identify, analyze, and evaluate risks associated with a proposed change and determine appropriate risk control measures.

**Change Classification**: Categorization of changes based on their potential impact on product quality, safety, efficacy, and regulatory compliance.

**Change Control Committee (CCC)**: A cross-functional team responsible for reviewing and approving changes with significant potential impact.

**Emergency Change**: A change that requires immediate implementation to prevent or address a significant risk to product quality, patient safety, data integrity, or business continuity.

**Temporary Change**: A change that is implemented for a limited duration with a defined expiration date, after which the system, process, or document reverts to its previous state.

**Revalidation**: The process of repeating validation activities to ensure that changes to a system, process, or method have not adversely affected its validated state.

**Post-Implementation Review**: An evaluation conducted after a change has been implemented to verify that the change was implemented as approved and is functioning as intended.

**Change Implementation**: The process of executing the approved change according to the defined implementation plan.

**Change Closure**: The formal completion of a change request after successful implementation and post-implementation review.

## 3. Change Classification

### 3.1 Change Types

Changes are categorized by type to facilitate appropriate evaluation and control:

#### 3.1.1 Facility Changes

- Modifications to building structures
- HVAC system changes
- Utility system changes (water, gas, electricity)
- Environmental monitoring system changes
- Layout or flow changes
- Classification changes to controlled areas

#### 3.1.2 Equipment Changes

- New equipment installation
- Equipment replacement or relocation
- Equipment modifications
- Equipment control system changes
- Calibration method changes
- Maintenance procedure changes

#### 3.1.3 Process Changes

- Manufacturing process changes
- Process parameter changes
- In-process control changes
- Batch size changes
- Manufacturing time or sequence changes
- Process automation changes

#### 3.1.4 Material Changes

- Raw material supplier changes
- Raw material specification changes
- Component changes
- Packaging material changes
- Cleaning agent changes
- Process aid changes

#### 3.1.5 Method Changes

- Analytical method changes
- Testing procedure changes
- Sampling plan changes
- Reference standard changes
- Stability protocol changes
- Laboratory equipment changes

#### 3.1.6 Documentation Changes

- SOP revisions
- Specification updates
- Batch record modifications
- Validation protocol changes
- Quality system document changes
- Form template changes

#### 3.1.7 Organizational Changes

- Key personnel changes
- Reporting structure changes
- Outsourcing or insourcing activities
- Significant staffing level changes
- Changes to quality responsibilities

#### 3.1.8 Computerized System Changes

- Hardware changes
- Software changes
- Configuration changes
- Interface changes
- Data management changes
- Security control changes

### 3.2 Change Classification Levels

Changes are classified by impact level to determine the appropriate review, approval, and implementation requirements:

#### 3.2.1 Level 1 (Minor Change)

- No direct impact on product quality, safety, or efficacy
- No significant impact on validated systems
- No impact on regulatory filings or commitments
- Examples: Administrative updates, like-for-like replacements, clarifications to documents

#### 3.2.2 Level 2 (Moderate Change)

- Limited potential impact on product quality, safety, or efficacy
- Limited impact on validated systems
- No significant impact on regulatory filings or commitments
- Examples: Minor process parameter adjustments, minor equipment modifications, updates to SOPs

#### 3.2.3 Level 3 (Major Change)

- Significant potential impact on product quality, safety, or efficacy
- Significant impact on validated systems
- May require regulatory notification or minor filing updates
- Examples: Process changes, equipment changes, specification changes, analytical method changes

#### 3.2.4 Level 4 (Critical Change)

- Major potential impact on product quality, safety, or efficacy
- Major impact on validated systems
- Likely requires regulatory approval prior to implementation
- Examples: Major manufacturing process changes, formulation changes, major facility modifications

### 3.3 Change Classification Process

The classification process consists of the following steps:

1. **Initial Classification**
   - The Change Initiator proposes an initial classification based on the change type and perceived impact
   - The Change Owner reviews and confirms or adjusts the initial classification

2. **Quality Assessment**
   - Quality Assurance reviews the proposed classification
   - Quality Assurance may upgrade or downgrade the classification based on quality impact assessment

3. **Regulatory Assessment**
   - Regulatory Affairs reviews the classification for regulatory impact
   - Regulatory Affairs may upgrade the classification based on regulatory filing requirements

4. **Final Classification**
   - The final classification is documented in the change request
   - The classification determines the review, approval, documentation, and implementation requirements

**Classification Decision Tree:**

```
Is there a direct impact on product quality, safety, or efficacy?
  |
  +-- NO: Is there an impact on validated systems?
  |       |
  |       +-- NO: Is there an impact on regulatory filings?
  |       |       |
  |       |       +-- NO: Level 1 (Minor Change)
  |       |       |
  |       |       +-- YES: Level 2 (Moderate Change)
  |       |
  |       +-- YES: Is the impact significant?
  |               |
  |               +-- NO: Level 2 (Moderate Change)
  |               |
  |               +-- YES: Level 3 (Major Change)
  |
  +-- YES: Is the impact significant?
          |
          +-- NO: Is there a regulatory filing impact?
          |       |
          |       +-- NO: Level 2 (Moderate Change)
          |       |
          |       +-- YES: Level 3 (Major Change)
          |
          +-- YES: Does it require prior regulatory approval?
                  |
                  +-- NO: Level 3 (Major Change)
                  |
                  +-- YES: Level 4 (Critical Change)
```

## 4. Change Control Process

### 4.1 Change Request Initiation

#### 4.1.1 Change Request Form

All changes must be documented using the approved Change Request Form (see Appendix A), which includes:

- Change requester information
- Change description and justification
- Current state and proposed state
- Business and/or technical justification
- Proposed implementation timeline
- Initial impact assessment
- Proposed change classification
- Supporting documentation

#### 4.1.2 Change Request Submission

The completed Change Request Form is submitted to:

- Department Head for initial review
- Change Control Coordinator for logging and tracking
- Quality Assurance for compliance review

#### 4.1.3 Change Request Tracking

All change requests will be:

- Assigned a unique tracking number (format: CC-[YEAR]-[SEQUENTIAL NUMBER])
- Recorded in the Change Control Log
- Tracked through to completion or rejection
- Reported on in regular change management metrics

### 4.2 Initial Assessment

The initial assessment includes:

1. **Completeness Check**
   - Verify all required information is provided
   - Verify appropriate justification is provided
   - Verify supporting documentation is attached

2. **Change Classification**
   - Determine the change type
   - Determine the change impact level
   - Document the classification rationale

3. **Change Owner Assignment**
   - Assign an appropriate Change Owner based on the change type and department
   - Notify the assigned Change Owner

4. **Resource Assessment**
   - Preliminary identification of required resources
   - Preliminary identification of required expertise
   - Preliminary timeline assessment

### 4.3 Impact Assessment

A comprehensive impact assessment must be conducted for all changes. The impact assessment:

1. **Identifies affected areas:**
   - Products
   - Processes
   - Equipment
   - Facilities
   - Materials
   - Documentation
   - Systems
   - Personnel

2. **Evaluates potential impacts on:**
   - Product quality, safety, and efficacy
   - Process performance and capability
   - Validated state of systems
   - Regulatory compliance
   - GxP documentation
   - Training requirements
   - Business operations

3. **Documents the assessment using:**
   - Impact Assessment Form (see Appendix B)
   - Supporting analysis documentation
   - Subject matter expert input

Detailed impact assessment requirements are provided in Section 5.

### 4.4 Risk Assessment

A risk assessment must be performed for all Level 2, 3, and 4 changes to:

1. **Identify potential risks associated with:**
   - The change implementation
   - The changed state
   - Potential failure modes

2. **Evaluate risks in terms of:**
   - Severity of consequences
   - Probability of occurrence
   - Detectability

3. **Determine risk control measures:**
   - Preventive actions
   - Detective actions
   - Mitigation strategies

4. **Evaluate residual risk:**
   - Assess remaining risk after controls
   - Determine acceptability
   - Determine monitoring requirements

Detailed risk assessment methodology is provided in Section 6.

### 4.5 Review and Approval

The review and approval process varies based on the change classification:

#### 4.5.1 Level 1 (Minor) Change Approval

- Department Head review and approval
- Quality Assurance review and approval
- Documentation in change control system

#### 4.5.2 Level 2 (Moderate) Change Approval

- Department Head review and approval
- Quality Assurance review and approval
- Regulatory Affairs review (if applicable)
- Validation review (if applicable)
- Documentation in change control system

#### 4.5.3 Level 3 (Major) Change Approval

- Department Head review and approval
- Quality Assurance review and approval
- Regulatory Affairs review and approval
- Validation review and approval
- Change Control Committee review and approval
- Documentation in change control system

#### 4.5.4 Level 4 (Critical) Change Approval

- Department Head review and approval
- Quality Assurance review and approval
- Regulatory Affairs review and approval
- Validation review and approval
- Change Control Committee review and approval
- Senior Management review and approval
- Documentation in change control system

Detailed review and approval requirements are provided in Section 7.

### 4.6 Implementation Planning

Following approval, an implementation plan must be developed that includes:

1. **Implementation steps and schedule**
   - Detailed action items
   - Timeline
   - Dependencies
   - Milestones

2. **Required resources**
   - Personnel
   - Materials
   - Equipment
   - Systems

3. **Validation and verification activities**
   - Testing requirements
   - Validation protocols
   - Acceptance criteria

4. **Training requirements**
   - Personnel to be trained
   - Training content
   - Training schedule

5. **Communication plan**
   - Stakeholders to be informed
   - Communication methods
   - Communication schedule

6. **Contingency and rollback plan**
   - Triggers for contingency actions
   - Rollback procedures
   - Decision authority

Detailed implementation planning requirements are provided in Section 8.

### 4.7 Implementation

The implementation phase includes:

1. **Pre-implementation verification**
   - Verify all prerequisites are met
   - Verify all approvals are obtained
   - Verify training is completed
   - Verify documentation is ready

2. **Change execution**
   - Execute according to approved implementation plan
   - Document all implementation activities
   - Record any deviations from the plan
   - Collect implementation evidence

3. **Post-implementation verification**
   - Verify the change was implemented as approved
   - Verify the change meets acceptance criteria
   - Document verification results

4. **Implementation documentation**
   - Complete Change Implementation Form (see Appendix D)
   - Attach implementation evidence
   - Document any deviations and their resolution

### 4.8 Post-Implementation Monitoring

Post-implementation monitoring includes:

1. **Monitoring activities**
   - Defined monitoring parameters
   - Monitoring frequency
   - Data collection methods
   - Alert criteria

2. **Effectiveness evaluation**
   - Evaluate if the change achieved its objectives
   - Evaluate if there are any unintended consequences
   - Document effectiveness assessment

3. **Post-implementation review**
   - Conduct formal review after a defined period
   - Evaluate monitoring data
   - Document lessons learned
   - Identify any follow-up actions

Detailed post-implementation monitoring requirements are provided in Section 10.

### 4.9 Change Closure

The change closure process includes:

1. **Closure requirements**
   - All implementation activities completed
   - All documentation updated
   - All training completed
   - Post-implementation verification completed
   - Effectiveness evaluation completed

2. **Closure approval**
   - Change Owner recommendation for closure
   - Quality Assurance approval for closure

3. **Closure documentation**
   - Update change status to "Closed"
   - Document closure date
   - Document closure justification
   - Archive change documentation

## 6. Risk Assessment Methodology

### 6.1 Risk Assessment Process

The risk assessment process follows ICH Q9 principles and includes:

#### 6.1.1 Risk Identification

- Structured identification of potential risks
- Use of appropriate tools (e.g., FMEA, FTA, HAZOP)
- Consideration of historical data and experience
- Input from subject matter experts

#### 6.1.2 Risk Analysis

- Evaluation of severity, probability, and detectability
- Qualitative or quantitative analysis methods
- Determination of risk levels
- Documentation of analysis results

#### 6.1.3 Risk Evaluation

- Comparison of risk levels to acceptance criteria
- Prioritization of risks for control measures
- Determination of risks requiring mitigation
- Documentation of evaluation results

#### 6.1.4 Risk Control

- Development of risk control measures
- Evaluation of control measure effectiveness
- Implementation of control measures
- Documentation of control measures

#### 6.1.5 Risk Review

- Periodic review of residual risks
- Monitoring of risk control effectiveness
- Update of risk assessment as needed
- Documentation of review results

### 6.2 Risk Evaluation Criteria

Risk evaluation uses the following criteria:

#### 6.2.1 Severity

| Level | Description | Examples |
|-------|-------------|----------|
| 1 | Negligible | No impact on product quality, safety, or efficacy |
| 2 | Minor | Minor impact on product quality; no impact on safety or efficacy |
| 3 | Moderate | Moderate impact on product quality; minor impact on safety or efficacy |
| 4 | Major | Significant impact on product quality; moderate impact on safety or efficacy |
| 5 | Critical | Direct impact on product safety or efficacy |

#### 6.2.2 Probability

| Level | Description | Likelihood |
|-------|-------------|------------|
| 1 | Remote | Unlikely to occur (less than once in 5 years) |
| 2 | Unlikely | May occur occasionally (once in 2-5 years) |
| 3 | Possible | Likely to occur (once in 1-2 years) |
| 4 | Likely | Likely to occur multiple times (several times per year) |
| 5 | Almost Certain | Expected to occur frequently (monthly or more often) |

#### 6.2.3 Detectability

| Level | Description | Detection Capability |
|-------|-------------|----------------------|
| 1 | Certain | Will definitely be detected before product release |
| 2 | High | High probability of detection before product release |
| 3 | Moderate | Moderate probability of detection before product release |
| 4 | Low | Low probability of detection before product release |
| 5 | Remote | Unlikely to be detected before product release |

#### 6.2.4 Risk Priority Number (RPN)

RPN = Severity × Probability × Detectability

RPN ranges are categorized as follows:
- Low Risk: RPN < 20
- Medium Risk: RPN 20-40
- High Risk: RPN 41-80
- Critical Risk: RPN > 80

### 6.3 Risk Control Measures

Risk control measures should follow this hierarchy:

#### 6.3.1 Risk Elimination

- Redesign the process to eliminate the risk
- Change materials or methods to eliminate the risk
- Use alternative approaches that don't present the risk

#### 6.3.2 Risk Reduction

- Modify the process to reduce probability or severity
- Add redundancy to critical systems
- Implement additional controls to reduce risk

#### 6.3.3 Risk Detection

- Implement in-process controls
- Add monitoring systems
- Increase testing frequency
- Enhance analytical methods

#### 6.3.4 Risk Mitigation

- Develop contingency plans
- Prepare emergency procedures
- Establish response protocols

### 6.4 Residual Risk Evaluation

After implementing control measures, residual risk must be evaluated:

#### 6.4.1 Residual Risk Calculation

- Recalculate severity, probability, and detectability
- Calculate new RPN
- Document residual risk level

#### 6.4.2 Residual Risk Acceptance

- Define acceptance criteria for residual risk
- Document acceptance rationale
- Obtain appropriate approvals for accepted risks

#### 6.4.3 Residual Risk Monitoring

- Define monitoring requirements
- Establish alert criteria
- Document monitoring procedures
- Assign monitoring responsibilities
